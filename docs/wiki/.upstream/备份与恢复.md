# 备份与恢复

## 备份内容
需要明确两件事：
1. 备份直接备份数据库内容。
2. 密码库条目本身在数据库里就是客户端加密后的状态。
所以没有再设计一层“备份包二次加密”功能。代码里实际做的是把数据库行直接写进 `db.json`，然后打包成 ZIP。

## 备份包结构

备份 ZIP 至少包含两个文件：

- `manifest.json`
- `db.json`

恢复时也会先检查这两个文件是否存在；缺任意一个都会直接报错。

`manifest.json` 记录的是备份元信息，包括：

- 格式版本
- 导出时间
- 应用版本
- 当前附件存储类型
- 各表记录数量
- 是否包含附件
- 附件文件数、总大小、最大对象大小

`db.json` 记录的是实际数据库行数据，包括：
- `config`
- `users`
- `user_revisions`
- `folders`
- `ciphers`
- `attachments`

## 勾选“包含附件”后会发生什么

勾选“包含附件”后，备份会同时处理两部分内容：

- `db.json` 中的附件记录
- 实际附件文件

`manifest.json` 会记录附件清单，包含：

- `cipherId`
- `attachmentId`
- `blobName`
- `sizeBytes`

如果包含附件，附件不会塞进数据库 JSON，而是作为独立文件参与备份和恢复。代码里要求每条附件记录都要能对应到 `attachments/<cipherId>/<attachmentId>.bin` 这一类文件路径，缺失时恢复会报错或进入跳过逻辑。

远程备份模式下，附件会单独保存在远程目录下的 `attachments/` 中；恢复时会按 `attachments/<blobName>` 去取回文件。

## 备份文件名校验

备份 ZIP 文件名带有基于文件内容计算出来的哈希前缀。导出时先打包 ZIP，再计算 SHA-256 前缀并写入文件名。

远程恢复时会先校验这个前缀是否和文件内容匹配；默认不允许跳过校验。校验不通过时，恢复会直接返回错误。

## 数据还原

远程恢复点击后，实际流程大致如下：

1. 读取远程 ZIP 备份文件。
2. 校验文件名中的哈希前缀是否和文件内容匹配。
3. 解析 `manifest.json` 和 `db.json`。
4. 检查目标实例是否为空；如果不是空实例，则根据 `replaceExisting` 决定继续还是报错。
5. 创建影子表。
6. 先把备份里的数据库数据导入影子表。
7. 校验影子表中的记录数量。
8. 按需恢复附件文件。
9. 对恢复失败的附件，删除对应附件行，避免留下脏记录。
10. 将影子表切换为正式表。
11. 如果是覆盖恢复，再清理当前实例中已经失去引用的旧附件对象。

## 点击“还原”后的实际结果

点击“还原”后，结果不是“合并现有数据与备份数据”，而是下面这种行为：

- 数据库内容以所选备份为准
- 如果开启覆盖恢复，现有实例数据会被替换
- 附件会在数据库导入后再逐个恢复
- 恢复失败的附件不会保留对应数据库行

所以最终状态应当理解为：

- 恢复成功的数据会完整进入实例
- 恢复失败的附件会被跳过
- 数据库中不会保留指向缺失附件的脏引用

## 附件恢复失败时会发生什么

远程恢复时，系统会先恢复数据库里的附件记录，再尝试恢复实际附件文件。对于没有成功恢复的附件，系统会把这些附件行从数据库里移除。

恢复结果里也会返回：

- 成功导入的附件数量
- 成功恢复的附件文件数量
- 被跳过的附件数量
- 被跳过原因
